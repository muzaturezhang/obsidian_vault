## 1. 我当前角色重定义：Agent Experiment Reviewer / Research Supervisor

你今天**不是**要亲自把 LoRA、Adapter、condition renderer、training script 全部搞懂，也不是要自己重写训练代码。

你今天真正负责的是：

> **审查 agent 产出的实验是否可信，并把“下一轮该做什么”写成具体 directive。**

你的最小可交付成果不是“我完全懂项目了”，而是：

> **一份 1 页审查记录 + 下一轮 agent 指令。**

你今天只需要看这些证据：

|你要确认什么|看什么|
|---|---|
|数据清洗是否真的发生|cleaned data manifest / 数据数量变化 / 被过滤 case 列表|
|训练是否用了 cleaned data|train config / command / log 里的 data path|
|LoRA + Adapter 是否真的训练并保存|checkpoint 路径 / step / 时间戳 / config|
|inference 是否加载了新 checkpoint|inference log / model path / output metadata|
|GT / base / ours / shuffled 是否公平|seed / case id / condition path / prompt / frame count|
|ours 是否有控制迹象|对比视频：人物、视角、地图结构、时序|
|agent 结论是否可信|它有没有列证据，而不是一句“looks better”|

工程上，ML 实验最怕的不是 loss 不好，而是**实验不可追踪**：数据、配置、checkpoint、seed、输出 artifact 没有被清楚记录。可复现实验通常要求记录数据、代码、配置、环境、输出，并用版本化 artifact 串起来；自动化 logging 的核心价值就是让每次实验的参数、指标、artifact、上下文可回查。([Wikipedia](https://en.wikipedia.org/wiki/Reproducibility?utm_source=chatgpt.com "Reproducibility"))

## 2. Quick Background for Beginners

### 2.1 world model 在这个项目里大概是什么

你们这里的 world model 可以先理解为：**给模型一些关于游戏世界状态的条件，让它生成更符合真实场景的视频/未来帧**。它不是单纯“生成好看的视频”，而是要让生成结果服从地图、玩家位置、视角、动态状态等条件。今天你不用深究完整 world model 理论，只要看：生成视频是否比 base 更接近 GT，尤其是人物位置、视角、场景结构、时序是否更受 condition 控制。

### 2.2 LoRA 是什么，今天需要关心什么

LoRA 是一种参数高效微调方法：冻结原始大模型，只训练小的低秩增量参数，从而低成本适配新任务。LoRA 原论文的核心就是冻结预训练权重，并注入可训练的低秩矩阵，减少可训练参数和显存需求。([arXiv](https://arxiv.org/abs/2106.09685?utm_source=chatgpt.com "LoRA: Low-Rank Adaptation of Large Language Models")) 今天你不用懂 LoRA 数学，只要关心：**本轮 LoRA checkpoint 是否真的重新训练、是否加载、是否对应 cleaned data，而不是旧模型冒充新结果。**

### 2.3 Adapter 是什么，今天需要关心什么

Adapter 可以先理解为：**把外部条件信号接到大模型里的小模块**。在条件生成里，adapter 常用于把 depth、pose、mask、map、玩家状态等条件转成模型能用的控制信息。T2I-Adapter 这类方法的思想就是冻结原始生成模型，训练轻量 adapter 来对齐外部控制信号。([arXiv](https://arxiv.org/abs/2302.08453?utm_source=chatgpt.com "T2I-Adapter: Learning Adapters to Dig out More Controllable Ability for Text-to-Image Diffusion Models")) 今天你不用懂 adapter 架构，只要看：**加了 Adapter 后，ours 是否更服从 condition。**

### 2.4 dynamic data / cleaned data 可能指什么

dynamic data 大概率指和时间变化有关的数据：玩家位置、朝向、动作、视角、alive 状态、敌我状态、轨迹、帧级 metadata 等。cleaned data 可能是过滤掉坏轨迹、错位帧、异常 episode、无效状态、motion/视频不同步的数据。今天你要关心的不是清洗算法多优雅，而是：**清洗前后样本数量、case id、过滤原因、训练实际 data path 是否一致。**

### 2.5 GT / base / ours / shuffled 分别代表什么

GT 是真实视频或真实帧，是参考答案。base 是不加你们新控制模块或不加 adapter 的基础生成结果。ours 是加了你们 LoRA + Adapter / condition control 后的结果。shuffled 通常是把 condition 打乱后生成的负对照：如果 ours 真用了 condition，shuffled 应该变差。今天最重要的是：**四者必须同 case、同 seed、同输入长度，否则不能比较。**

### 2.6 fixed seed comparison 为什么重要

生成模型有随机性。同一个模型因为 seed 不同，可能生成质量、人物位置、画面细节都不同。fixed seed 的意义是：**尽量让 base 和 ours 的随机噪声一致，把差异更多归因到模型/condition，而不是运气。**但只固定一个 seed 也不够证明结论，最多说明这个 seed 下可比。更稳的是多个固定 seed + 多个 case。随机种子能帮助复现，但严肃实验也要避免只在单个 seed 上下结论。([Medium](https://medium.com/data-science/random-seeds-and-reproducibility-933da79446e3?utm_source=chatgpt.com "Random Seeds and Reproducibility"))

### 2.7 什么叫“ours 有控制迹象”

不是“ours 看起来更清晰”，而是：**ours 相比 base 更符合输入 condition 或更接近 GT 中的关键可控因素**。比如同一时刻人物出现在更接近 GT 的位置，camera/viewpoint 更对，地图结构更稳定，shuffled 退化明显。最强证据是：多个 case 中 ours 都比 base 更对，而且 shuffled 变差，说明 condition 不是摆设。

### 2.8 什么叫“失败 case”

失败 case 不是垃圾结果，而是**暴露系统问题的结果**。例如：ours 和 base 差不多；ours 比 base 更差；人物位置完全错；condition 被忽略；画质变好但控制没变；视频前几帧正常后面崩；同一场景变成另一个场景；多视角之间互相矛盾。失败 case 很有价值，因为它告诉下一轮应该修数据、修对齐、修 adapter、修 inference，还是修评估。

### 2.9 为什么不能只看单个好看的视频

生成模型很容易 cherry-pick。一个好看的 case 可能来自随机 seed、简单场景、训练集记忆、视觉质量提升，而不是 condition control。视频生成评估里，时序一致性和动态控制都很关键，已有研究也指出视频模型常在长时依赖、内容连续性、动态可控性上失败。([arXiv](https://arxiv.org/abs/2210.02396?utm_source=chatgpt.com "Temporally Consistent Transformers for Video Generation")) 所以你今天要记录成功和失败 case，而不是只挑最漂亮的 demo。

### 2.10 为什么监督 agent 的核心不是写代码

agent 会写代码、跑实验、生成视频，但它很容易犯“路径错了但看似成功”“加载旧 checkpoint”“结论没证据”“只展示好 case”的错误。你的核心价值是：**判断证据链是否闭合**。也就是从 data → train config → checkpoint → inference config → video artifact → conclusion，每一步是否能对上。

## 4. 今天 3 小时执行计划

### Hour 0–1: Minimum Knowledge Bootstrapping

**目标：**  
只补够你能审查结果的知识，不补完整世界模型。

**必须读/问/确认什么：**

1. 当前最新 run 在哪里：  
    `run_id / output_dir / report.md / videos / config / checkpoint / ledger`
    
2. 本轮实验目标一句话：  
    “cleaned dynamic data 是否让 LoRA + Adapter 更有 condition control？”
    
3. 四个对比的定义：  
    GT / base / ours / shuffled 分别是什么。
    
4. shuffled 到底 shuffle 了什么：  
    shuffle condition？shuffle time？shuffle case？shuffle player state？
    
5. 当前有效 case 数量：  
    生成了几个 case，几个 seed。
    
6. cleaned data 的证据：  
    清洗前后数量、过滤规则、manifest path。
    
7. 本轮 checkpoint 的证据：  
    checkpoint path、step、训练时间、是否被 inference 加载。
    

**不用深入理解什么：**

- LoRA 的矩阵分解；
    
- Adapter 的内部层结构；
    
- DiT latent token 细节；
    
- loss 的数学定义；
    
- condition renderer 的完整实现；
    
- 视频生成 diffusion sampling 原理。
    

**结束时你应该能回答：**

- 这轮实验到底想证明什么？
    
- 哪个 run 是最新结果？
    
- 哪些视频是公平对比？
    
- cleaned data 是否真的进入训练？
    
- inference 是否真的加载新 checkpoint？
    
- 我接下来该看哪 3–5 个 case？

### Hour 1–2: Result Inspection

**目标：**  
打开最新实验结果，判断“是否有控制迹象”，并找出明显 pipeline 问题。

**优先打开：**

1. 最新 run 的 `summary/report.md`；
    
2. `experiment_ledger.jsonl` 或等价 run ledger；
    
3. train config；
    
4. inference config；
    
5. checkpoint directory；
    
6. GT/base/ours/shuffled 拼接视频；
    
7. 每个视频对应 metadata JSON。
    

**每个结果看什么：**

|结果|你看什么|
|---|---|
|report.md|agent 有没有具体 case 证据|
|train config|data path 是否 cleaned|
|checkpoint|是否新生成、step 是否合理|
|inference config|是否加载新 checkpoint|
|video metadata|seed/case/frame 是否一致|
|GT/base/ours/shuffled|ours 是否更接近 GT，shuffled 是否退化|
|failed cases|失败类型是否被记录|

**视频观看顺序：**

先不要全看。看 3 类 case：

1. agent 声称最成功的 1–2 个；
    
2. agent 声称失败的 1–2 个；
    
3. 随机抽 1–2 个没有被重点展示的 case。
    

**如何判断是否有效：**

你可以用一句话判断：

> 如果 ours 相比 base 更接近 GT 的“条件相关内容”，且 shuffled 明显更差，那么这是控制迹象。  
> 如果 ours 只是更清晰、更像游戏，但人物/视角/位置不更对，那不是强证据。

## 5. 结果审查 checklist

你看每个 GT / base / ours / shuffled 对比视频时，按这个打勾：

### A. 公平性检查

|检查项|具体怎么查|
|---|---|
|同一 seed|文件名或 metadata 是否写 seed；base/ours/shuffled 是否一致|
|同一 case|episode id、player id、map、frame_start 是否一致|
|同一输入条件|condition path 是否相同；shuffled 除外，但要知道 shuffle 了什么|
|同一视频长度|四个 panel 的 frame count 是否一致|
|同一采样设置|guidance、steps、resolution、fps 是否一致|
|同一 checkpoint 类型|base 是 base，ours 是新 ckpt，不要混旧 ckpt|
|GT 对齐|GT 是否对应同一时间段，不是随便拼的真实视频|

只要 A 部分有问题，这个 case 标为：

> **invalid comparison，不用于结论。**

---

### B. 控制效果检查

|检查项|具体怎么看|
|---|---|
|camera/viewpoint 是否更对|ours 的朝向、转弯、视角变化是否比 base 更接近 GT|
|scene layout 是否更对|墙、门、通道、室内外关系是否更像 GT|
|player/object 位置是否更对|人物是否出现在 GT 对应区域，而不是随机出现|
|motion 是否更对|运动方向、速度、停顿是否更接近 GT|
|ours 是否优于 base|不看美观，看 condition-relevant 内容|
|ours 是否优于 shuffled|如果 shuffled 差不多，说明 condition 可能没被用上|
|是否跨帧稳定|前后帧人物/场景是否连续，不突然跳变|
|是否多视角一致|如果有多视角，人物/空间关系是否互相矛盾|

---

### C. 假进步检查

|假进步|具体表现|
|---|---|
|只改善画质|ours 更清晰，但位置/视角/人物不更对|
|只改善颜色/纹理|地图结构和控制无变化|
|随机变好|只有一个 seed 好，换 seed 不稳定|
|简单 case 变好|只在静止、无人物、短视频中好|
|shuffled 也好|打乱 condition 后也差不多，说明不是 condition 控制|
|训练集记忆|视频像某个训练片段，但和当前 condition 不匹配|
|前几帧好后面崩|短期对齐，长期时序失败|

---

### D. failure case 记录

每个失败 case 至少记录：

```text
case_id:
seed:
failure_type:
where_it_starts:
GT observation:
base observation:
ours observation:
shuffled observation:
likely cause:
next action:
```

failure_type 建议用这些标签：

- `invalid_comparison`
    
- `no_control_signal`
    
- `weak_control_signal`
    
- `shuffled_not_degraded`
    
- `viewpoint_mismatch`
    
- `player_position_wrong`
    
- `scene_layout_drift`
    
- `temporal_degradation`
    
- `multi_view_inconsistent`
    
- `quality_only_improvement`
    
- `pipeline_suspected_bug`

## 6. “ours 有控制迹象”的判定标准

|标签|判定标准|你看视频时怎么写|
|---|---|---|
|**Strong control**|ours 明显比 base 更接近 GT；关键人物/视角/场景结构更对；shuffled 明显退化；多个 case/seed 重复出现|“ours 在人物位置和视角变化上均优于 base，shuffled 退化，说明 condition 被使用。”|
|**Moderate control**|ours 在主要区域更接近 GT，但仍有明显错误；shuffled 有一定退化；不是所有帧都稳定|“有局部控制迹象，但时序/人物位置仍不稳定。”|
|**Weak control**|个别帧或局部区域更好，但整体不稳定；可能只是随机改善|“只看到弱迹象，需要更多 seed/case 验证。”|
|**No control**|ours 与 base 差不多；shuffled 与 ours 差不多；改善主要是画质不是对齐|“没有看到 condition-specific improvement。”|
|**Failure**|ours 更差；视角/人物/场景错；时序崩；多视角关系不成立|“ours 未服从 condition，属于失败 case。”|
|**Invalid**|seed/case/frame/checkpoint 不一致，不能比较|“对比不公平，不能用于结论。”|

一句最实用的判断：

> **看 ours 有没有更“听 condition 的话”，不要看它有没有更好看。**
